# Topo v0.9.4

Topo v0.9.4 is a package lifecycle hardening release. It makes `topo update` and `topo remove` work directly for GitHub Release `.deb` and `.rpm` installs, fixes shell-cache edge cases after switching install methods, and improves the reliability of release update checks.

## Highlights

### Unified Lifecycle Commands

Package installs now use the same user-facing lifecycle commands as script installs:

```bash
topo update
topo remove
```

For `.deb` / `.rpm` installs, `topo update` now:

- reads the local installed version
- checks the latest GitHub Release
- chooses the matching package for the current distro and architecture
- downloads the `.deb` or `.rpm`
- downloads `SHA256SUMS`
- verifies the package checksum
- upgrades through `apt` or `dnf`

For `.deb` / `.rpm` installs, `topo remove` now:

- removes the package through `apt` or `dnf`
- cleans Topo user config/cache/state for the current user
- removes script-install remnants when present
- removes Topo-managed launcher entries
- keeps package-owned files under package-manager control

### Package Install Compatibility Launcher

Package post-install scripts now create a managed compatibility launcher for the invoking `sudo` user when possible:

```text
~/.local/bin/topo
```

This solves the common migration case where a shell has cached an older script-install command path before a package install creates `/usr/bin/topo`. After package installation, `topo` works immediately even in that existing shell session.

The compatibility launcher:

- is created only for the invoking non-root user when detectable
- replaces missing, symlinked, or Topo-managed launchers
- does not overwrite arbitrary user-created regular files
- forwards to `/usr/bin/topo` when the package is installed
- falls back to `~/.topo/topo` if a script install exists

Package post-remove scripts clean the managed launcher when it is no longer needed.

### Cleaner Package Removal

Package removal now cleans both package-side and user-side residue more completely:

- empty `/usr/lib/topo` directory trees are removed by package post-remove scripts
- Topo-managed `~/.local/bin/topo` compatibility launchers are removed
- package-mode `topo remove` removes:
  - `~/.config/topo`
  - `~/.cache/topo`
  - `~/.local/state/topo`
  - `~/.topo`
  - Topo-managed launcher entries
  - Topo PATH blocks added to shell rc files

Direct package-manager removal still avoids deleting arbitrary user files. For full Topo cleanup, use:

```bash
topo remove
```

### Update Reliability

`topo update` is now more resilient when checking the latest GitHub Release:

- GitHub API requests include `User-Agent` and GitHub API `Accept` headers.
- If the GitHub API returns an error such as `403`, Topo falls back to resolving the `/releases/latest` redirect URL.
- Release asset downloads also include a `User-Agent`.

This avoids update failures caused by anonymous GitHub API throttling or API-side request rejection.

### Package Dependencies

Debian and RPM packages now explicitly depend on:

```text
curl
python3
python3-packaging
```

`curl` is required for package-mode `topo update` to check releases and download assets.

### Release Assets

Expected user-facing Release assets:

```text
topo_0.9.4_amd64.deb
topo_0.9.4_arm64.deb
topo-0.9.4-1.x86_64.rpm
topo-0.9.4-1.aarch64.rpm
topo-core-x86_64
topo-core-aarch64
SHA256SUMS
SHA256SUMS.asc
topo-release-public.asc
```

## Notes

- `topo update` currently verifies downloaded package integrity with `SHA256SUMS`.
- Automatic GPG verification of `SHA256SUMS.asc` is still future work.
- DNF's `@commandline` OpenPGP warning is not solved by checksum verification; RPM package signing and a DNF repository are separate future work.
- First-party APT/DNF repositories should come after several stable GitHub Release package versions.
